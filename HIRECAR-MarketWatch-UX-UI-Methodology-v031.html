1. Executive Intent

MarketWatch exists to scale discipline before scale volume. It sets standards upstream, routes users into controlled outcomes, and keeps the system defensible when incidents, billing fog, custody ambiguity, and vendor drift occur.

DO

  • Use MarketWatch as the authority layer: clear signals, clear posture, clear routing.
  • Keep "Apply to Qualify" stable and above the fold.
  • Teach standards through content: what compliant looks like.

DON'T

  • Don't treat MarketWatch like a generic blog.
  • Don't publish content noise that competes with conversion.
  • Don't weaken proof surfaces to gain clicks.

2. The HIRECAR Method (How UX/UI is governed)

2.1 Evidence Velocity Doctrine (EVD)

Time destroys evidence. Evidence protects outcomes. Any action that can create a dispute must create a record.

DO
Design "receipt-first": confirmations, timestamps, next steps, and support entry on every submission.
DON'T
Don't rely on email alone. Don't allow silent success. Don't overwrite receipts in redesigns.
2.2 Lane Routing as UI Posture

Revenue/Cure/Remedy posture determines friction and gating. Friction is a control lever, not a design flaw.

  • Revenue: scale-safe clarity and speed
  • Cure: controlled recovery with added confirmations
  • Remedy: containment posture with strict evidence intake
2.3 Access Governance

Permissions and restrictions are enforced through UX boundaries. Every restriction must tell the user: what, why, and what next.

2.4 Remedy Architecture

Recovery is productized through standard packs. Remedy workflows must collect structured evidence and generate export-ready packets.

3. MarketWatch v1.1 — Information Architecture

Non-negotiable home modules
Utility + Ticker · Masthead (Apply) · Hero (one) · Standards Strip · Feed (≤ 6) · Trending · Footer (support + policies).

DO

  • Limit homepage feed to six cards.
  • Maintain consistent card rhythm and metadata formatting.
  • Keep Apply stable across sessions and pages.

DON'T

  • Don't add modules without removing something else.
  • Don't introduce competing primary actions.
  • Don't move Trending above Hero on mobile.

4. Device Strategy — Mobile vs Desktop Execution

Mobile = Fast Signal + One Action

  • In 10 seconds: what is happening (ticker + one hero) and what to do next (Apply).
  • Keep modules short; reduce scrolling cost.
  • Keep Apply visible (top + repeated at high-intent points).

Desktop = Depth + Governance + Operator Mode

  • Two-rail layout: feed left, trending/signals right.
  • Provide deeper reference links (standards, "how it works").
  • No "dashboard cosplay" clutter.

5. Interaction Modes Standard — Desktop / Remote (Replaces Day/Night)

Core decision
Remove time-based switching (including hourly re-check). Desktop/Remote is intentional and user-controlled.

Desktop mode

  • Information-dense operator layout.
  • Two-rail on wide screens.
  • Apply stays above the fold.

Remote mode

  • Mobile-first visual menu hub (tiles).
  • Big tap targets route to existing anchors.
  • Bottom bar: Menu / Top / Switch Mode.
5.1 Mode Gate (First-run prompt)
  • Only appears if no preference is saved.
  • Desktop/Remote choice, plus "Remember on this device" (default ON).
  • Recommended highlight: Remote on mobile; Desktop on desktop.
5.2 Remote Home (Visual menu hub)

Remote mode adds a tile menu hub that routes to existing section IDs. It is designed to reduce mobile bounce by routing first and reading second.

Required tile targets
Credit Recovery (featured) · Top Stories · Trending · Mobility + Rentals · Entertainment · Services · Membership
5.3 CSS correctness gate (release blocker)

Fix invalid rgba alpha values site-wide. Alpha must be 0–1 (example: 0.12), not 12/65/5. This is a release blocker.

5.4 Mobile layout fixes (grid-first)

Where the layout uses grid containers, mobile corrections must update grid templates to one column—do not use flex-direction hacks on grids. Avoid competing sticky systems on mobile (nav sticky wins).

5.5 Persistence and Migration

If preference exists: restore immediately. If not: show Mode Gate. Migration rule: dark → Remote, light → Desktop. Silent one-time migration. Remove time-based switching from the background entirely.

6. Visual System Standards

Authority + Clean Enforcement

  • High contrast, strong hierarchy, minimal clutter.
  • CTA dominance in both modes.
  • Readable nav + footer in dark/remote.

Anti-patterns

  • Trend aesthetics that reduce readability.
  • Gray-on-gray text in dark/remote.
  • Multiple competing CTAs.

7. MarketWatch Page Specs (Module-by-module)

7.1 Utility Bar + Ticker

DO

  • Ticker is short, factual, standards-forward.
  • Readable in both Desktop and Remote.
  • Never louder than Apply.

DON'T

  • Don't use ticker for sales copy.
  • Don't imply approvals/guarantees.
7.2 Masthead (Apply gateway)

DO: Apply above the fold and stable. DON'T: hide Apply in menu-only navigation or rotate primary CTAs.

7.3 Hero

DO: one hero only, consistent metadata. DON'T: multiple featured zones.

7.4 Feed (≤ 6)

DO: one card pattern, stable ratios. DON'T: exceed six on homepage.

7.5 Trending

DO: ranked list; desktop right rail; mobile below feed. DON'T: trending above hero on mobile.

7.6 Proof surfaces (Receipts)

Every submission ends in a receipt surface containing: timestamp, summary, next steps, support entry, and reference ID (where applicable).

8. Engineering Handoff Standard (Wix + Velo)

8.1 Boot order stability
  • Initialize mode (with migration) before layout-dependent scripts.
  • Bind Mode Gate + toggle + Remote routing before counters/reveals/sticky detection.
  • Remove hourly time-based switching entirely.
8.2 Mobile layout discipline

Fix grid containers by updating grid templates; do not apply flex-direction hacks on grids. Prevent competing sticky systems.

8.3 CSS correctness gate

Invalid rgba alpha values must be corrected before release. Treat as a release blocker.

9. Contracts (IDs · CMS Schema · Tokens)

Contract rule
IDs and schema are contracts. Renaming breaks bindings. Token drift breaks consistency.
9.1 Element IDs

Assign element IDs exactly as the design contract requires. Treat renaming as a breaking change requiring coordination and approvals.

9.2 CMS schema

Design cards and modules to survive missing optional fields. Support "coming soon" editions with clear disabled states.

9.3 Tokens

Use tokens for colors and spacing. Document token changes as version updates. No random hex without tokens + documentation.

11. QA Gates (Release is controlled)

11.1 Must-pass checklist
  • Apply above fold (desktop + mobile)
  • Mode Gate only when no preference exists
  • Desktop/Remote toggle works + correct label
  • No time-based switching remains
  • Remote routing works + bottom bar respects safe area
  • Mobile grid stacking correct
  • Invalid rgba corrected
  • Receipt surfaces exist for submissions
11.2 Release blockers
  • Apply missing above fold
  • Mode switching breaks layout or hides content
  • Submissions without receipts
  • IDs/schema renamed after bind without coordinated release
  • Invalid rgba values remain
